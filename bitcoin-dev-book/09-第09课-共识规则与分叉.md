# 第 09 课 · 共识规则与分叉

> 全节点验证什么；软分叉与硬分叉的本质区别。
>
> **作者：曹宇春**

---

## 全节点验证清单

对新交易（mempool）：

- [ ] 语法正确、大小合法
- [ ] input 引用的 UTXO 存在且未花费
- [ ] 脚本验证通过（签名有效）
- [ ] Σinput ≥ Σoutput，无负值
- [ ] 不违反共识规则（locktime、sequence、weight…）

对新区块：

- [ ] 区块头 PoW 有效
- [ ] 第一笔为合法 coinbase，subsidy 不超过允许值
- [ ] 所有交易通过上述验证
- [ ] Merkle root 正确
- [ ] 时间戳、难度符合规则

**共识 = 所有全节点跑同一套代码规则。**

---

## 软分叉（Soft Fork）

**收紧规则**，新区块对旧节点仍「看起来合法」（或旧节点略宽松）。

- 旧节点可能认为新区块有效，但**不验证新语义**
- 需多数算力升级才能激活
- 例：P2SH（2012）、CLTV、SegWit（BIP-9/141）、Taproot

**激活机制：** BIP-9 版本位、Speedy Trial（Taproot）等。

---

## 硬分叉（Hard Fork）

**放宽或改变规则**，旧节点不认新区块 → **链分裂**。

- 例：2017 BCH（区块大小争议）、BSV
- 需生态各方协调，否则永久两条链、两种资产

---

## 常见分叉类型

| 类型 | 原因 | 持久性 |
|------|------|--------|
| 临时分叉 | 两矿工同时出块 | 下一块后自然解决 |
| 硬分叉 | 规则不兼容 | 永久分裂 |
| 软分叉 | 规则收紧 | 不分裂（成功时） |

---

## 用户视角

- **软分叉成功：** 用户无感，用升级后的钱包即可
- **硬分叉：** 持有私钥者在分叉点可能获得**两条链上的币**（若两边都有市场）

**运行全节点** = 自己投票认可哪条链的规则集。

---

## BIP 流程简述

```
Idea → BIP 草案 → 参考实现 PR → 社区讨论 → 矿工/节点升级 → 激活
```

重要 BIPs 存档：[github.com/bitcoin/bips](https://github.com/bitcoin/bips)

---

## 本课小结

| | 软分叉 | 硬分叉 |
|--|--------|--------|
| 规则 | 收紧 | 改变/放宽 |
| 兼容性 | 向前兼容 | 不兼容 |
| 链 | 通常一条 | 可能分裂 |
