# 第 01 课 · 比特币架构鸟瞰

> 建立技术全景，知道各层职责与数据流向。
>
> **作者：曹宇春**

---

## 系统分层

```
┌─────────────────────────────────────────┐
│  应用层：钱包、区块浏览器、交易所、闪电   │
├─────────────────────────────────────────┤
│  协议层：交易、Script、区块、共识规则     │
├─────────────────────────────────────────┤
│  网络层：P2P 传播、节点发现、内存池       │
├─────────────────────────────────────────┤
│  密码学层：SHA-256、RIPEMD-160、ECDSA     │
└─────────────────────────────────────────┘
```

比特币是**协议 + 参考实现（Bitcoin Core）**，不是某家公司提供的 API 服务。

---

## 核心状态：UTXO 集合

全节点维护的「当前世界状态」不是 `{地址: 余额}`，而是 **UTXO Set**：

```
UTXO = { (txid, vout_index) → (value, scriptPubKey) }
```

- **txid**：创建该输出的交易 ID
- **vout_index**：该交易第几个输出（从 0 开始）
- **value**：聪（satoshi），1 BTC = 10^8 sat
- **scriptPubKey**：锁定脚本，定义谁能花

每笔交易**消耗**若干 UTXO（input），**创建**若干 UTXO（output）。  
总输出（含手续费）≤ 总输入。

---

## 数据流向：一笔交易的一生

```
1. 钱包构造 raw transaction
2. 用私钥签名 input（scriptSig / witness）
3. 广播到 P2P 网络 → 进入各节点 mempool
4. 矿工从 mempool 选题打包进 block
5. PoW 出块 → 广播 block
6. 其他节点验证 block → 更新 UTXO set
7. 经过 N 个确认后，接收方认为最终性足够
```

---

## 关键参与者

| 角色 | 职责 |
|------|------|
| **全节点** | 下载区块、验证规则、维护 UTXO、转发交易 |
| **轻节点（SPV）** | 只存 block header，用 Merkle proof 验证交易 |
| **矿工** | 选题、PoW、出块、收 block reward + fee |
| **钱包** | 管理密钥、选 UTXO、构造/签名交易 |

---

## 与以太坊对比（避免思维惯性）

| | 比特币 | 以太坊 |
|--|--------|--------|
| 状态模型 | UTXO | 账户 + Storage |
| 程序 | Script（非图灵完备） | EVM（图灵完备） |
| 出块 | ~10 分钟 | ~12 秒 |
| 主开发点 | 交易、密钥、节点 | 合约、DApp |

# 第 02 课 · 哈希与数字签名

> 比特币用到的密码学 primitives，够用即可，不必推导证明。
>
> **作者：曹宇春**

---

## SHA-256：区块与交易的指纹

**性质（开发需要知道的）：**

- 固定长度输出（256 bit）
- 单向：已知 x 易算 hash(x)，已知 hash 难反推 x
- 抗碰撞：难找 x ≠ y 使得 hash(x) = hash(y)
- **雪崩效应**：输入改 1 bit，输出面目全非

**在比特币中的用途：**

| 用途 | 说明 |
|------|------|
| 区块哈希 | `hash(block_header)` 必须 < target |
| txid（legacy） | 对序列化交易 double SHA-256 |
| Merkle 树 | 叶子节点为交易哈希 |
| 地址 | `RIPEMD160(SHA256(pubkey))` |

```bash
# 命令行体验
echo -n "hello" | openssl dgst -sha256
```

**Double SHA-256：** 比特币中对交易、区块头常用 `SHA256(SHA256(x))`，记作 hash256。

---

## RIPEMD-160

公钥经 SHA-256 后再 RIPEMD-160，得到 20 字节，缩短地址长度。

```
pubkey (33/65 bytes)
    → SHA256 → 32 bytes
    → RIPEMD160 → 20 bytes (= pubkey hash)
```

---

## ECDSA 与 secp256k1

比特币使用 **椭圆曲线数字签名算法（ECDSA）**，曲线为 **secp256k1**（与以太坊相同曲线，格式细节不同）。

### 密钥对

- **私钥**：256 bit 随机数（严格说需在 [1, n-1] 范围内）
- **公钥**：私钥 × 基点 G（椭圆曲线点）
- **压缩公钥**：33 字节（0x02/0x03 + x 坐标）
- **非压缩公钥**：65 字节（0x04 + x + y）

### 签名

对交易摘要 `z` 用私钥签名，得到 `(r, s)`。  
验证方用公钥验证 `(r,s)` 与 `z` 是否匹配。

**Schnorr 签名**（BIP-340，Taproot 后）：更短、可批量验证、支持 MuSig 多签聚合。

### 交易中的意义

- **拥有私钥** = 能花费锁定到对应公钥/地址的 UTXO
- 签名放在 input 的 `scriptSig` 或 **witness**（SegWit）中

---

## 哈希指针 = 区块链接

区块头包含 `prev_block_hash`，即前一区块头的 hash256。

```
Block N-1  header  --hash-->  Block N header.prev_block_hash
```

改 Block N-1 的内容 → 其哈希变 → Block N 及之后全部无效（除非重做 PoW）。

---

## 动手：Python 生成密钥（理解用）

```python
# 需要 pip install ecdsa 或使用 bitcoinlib
# 仅演示概念，生产环境用成熟库
import hashlib
import os

privkey = os.urandom(32).hex()
print("private key (hex):", privkey)
```

生产环境请使用 **bitcoinlib**、**rust-bitcoin**、**bitcoinjs-lib**，不要自己实现密码学。

---

## 本课小结

| 算法 | 作用 |
|------|------|
| SHA-256 / hash256 | 区块 PoW、txid、Merkle |
| RIPEMD-160 | 压缩公钥 → 地址 |
| ECDSA secp256k1 | 交易授权 |
| Schnorr (BIP-340) | Taproot 默认签名 |

# 第 03 课 · Merkle 树与 SPV

> 区块如何承诺一组交易；轻节点如何只带区块头验证。
>
> **作者：曹宇春**

---

## Merkle 树结构

对区块内所有交易计算 txid，两两 hash 合并，逐层向上，直到单一根：

```
        Merkle Root
       /            \
    H(AB)          H(CD)
    /    \          /    \
  H(A)  H(B)     H(C)  H(D)
   |     |        |     |
  TxA   TxB      TxC   TxD
```

- 交易数为奇数时，最后一个节点**与自己配对**再 hash
- **Merkle Root** 写入区块头 `merkle_root` 字段
- 改任意一笔交易 → Root 变 → 区块头哈希变 → PoW 失效

---

## Merkle Proof（SPV 证明）

轻节点只有区块头，想确认交易 TxB 是否在区块中：

1. 节点提供 Merkle 路径：`H(A), H(CD), ...` 直到 Root
2. 轻节点本地计算：用 TxB 的 txid 与兄弟节点逐层 hash
3. 结果等于区块头中的 `merkle_root` → 交易确实在该块

**路径长度：** O(log N)，N 为交易数。

---

## SPV 钱包的权衡

| | 全节点 | SPV |
|--|--------|-----|
| 存储 | 完整区块 + UTXO | 仅 block headers |
| 验证 | 全部规则 | 交易在块中 + 最长链 |
| 信任 | 信任自己的验证 | 信任 peer 提供的 Merkle proof |
| 隐私 | 较好（可自建） | 需向 peer 泄露感兴趣地址 |

SPV **不验证**区块内所有交易是否合法，只验证**某笔交易是否被某矿工收录**。

---

## 默克尔树在比特币中的其他用途

- **见证 Merkle 树**（SegWit）：witness 数据单独成树
- **Taproot**：script path 用 Merkle 树隐藏分支脚本

---

## 代码直觉（伪代码）

```python
def merkle_root(txids: list[bytes]) -> bytes:
    layer = txids[:]
    while len(layer) > 1:
        if len(layer) % 2 == 1:
            layer.append(layer[-1])  # duplicate last
        layer = [hash256(layer[i] + layer[i+1])
                 for i in range(0, len(layer), 2)]
    return layer[0]
```

---

## 本课小结

- Merkle Root 是区块对交易列表的**紧凑承诺**
- SPV = 区块头 + Merkle proof，适合资源受限设备
- 全节点仍是最高安全模型

# 第 04 课 · UTXO 模型深入

> 开发者必须精通的账本模型。
>
> **作者：曹宇春**

---

## UTXO 精确定义

一个 UTXO 由 **OutPoint** 唯一标识：

```
OutPoint = (txid: 32 bytes, vout: uint32)
```

对应数据：

```json
{
  "value": 50000000,
  "scriptPubKey": "76a914...88ac",
  "confirmations": 6
}
```

`value` 单位是 **satoshi**（sat）。

---

## 交易平衡公式

```
Σ(input values) = Σ(output values) + fee
```

- **fee** 不写入任何 output，差额即矿工收入
- 没有「余额」字段；钱包余额 = 其可控 UTXO 的 value 之和

---

## 选币（Coin Selection）

钱包构造交易时需选择哪些 UTXO 作为 input：

| 策略 | 说明 |
|------|------|
| 最大优先 | 先花大额 UTXO，减少 input 数 |
| 最小碎片 | 合并 dust UTXO |
| Branch and Bound | 精确匹配目标金额，减少找零 |

差劲选币 → 多 input → 交易体积大 → 手续费高。

---

## 找零输出

几乎每笔支付都需要 **change output** 指回自己：

```
Input:  1.0 BTC (UTXO A)
Output: 0.3 BTC → 收款方
Output: 0.699 BTC → 自己新地址（找零）
Fee:    0.001 BTC
```

**隐私提示：** 重用找零地址会泄露关联；现代钱包倾向每次找零用新地址（HD 钱包）。

---

## Dust 限制

Bitcoin Core 默认拒绝过小的 output（受 `dustRelayFee` 影响），防止区块被极小 UTXO 塞满。

典型 dust 阈值约 **546 sat**（依赖 output 类型和费率）。

---

## UTXO Set 与性能

全节点维护**当前所有未花费输出**的集合（存在 Chainstate LevelDB 中）。

- 区块验证：检查 input 是否在 UTXO set 中、脚本是否执行成功、无双重花费
- 剪枝节点（pruned）：删旧区块数据，仍保留 UTXO set

---

## 与账户模型对比

| UTXO | 账户 |
|------|------|
| 并行花费不同 UTXO 较自然 | 同一账户 nonce 顺序 |
| 隐私性略好（可每次新地址） | 地址复用常见 |
| 交易体积随 input 数增长 | 固定大小转账 |
| 无全局余额状态 | 全局 state trie |

---

## 本课小结

- UTXO = `(txid, vout) → (value, scriptPubKey)`
- 交易必须平衡；差额是 fee
- 钱包核心能力之一是 **Coin Selection**

# 第 05 课 · 交易结构与生命周期

> 交易的字段、序列化、txid 与 wtxid。
>
> **作者：曹宇春**

---

## 交易结构概览

```text
Transaction
├── version          (4 bytes)
├── marker + flag      (SegWit only: 0x00 0x01)
├── inputs[]           (vin)
│   ├── prev_txid
│   ├── prev_vout
│   ├── scriptSig / (empty for segwit)
│   └── sequence
├── outputs[]          (vout)
│   ├── value
│   └── scriptPubKey
└── locktime
    └── witness[]      (SegWit, per input)
```

---

## Input（vin）

| 字段 | 说明 |
|------|------|
| `prev_txid` | 引用的 UTXO 所在交易（小端序显示） |
| `prev_vout` | 输出索引 |
| `scriptSig` | 解锁脚本（legacy）或空（SegWit） |
| `sequence` | 默认 0xFFFFFFFF；用于 RBF、相对时间锁 |

**sequence 与 RBF：** `sequence < 0xFFFFFFFE` 表示该交易可被更高费版本替换（BIP-125）。

---

## Output（vout）

| 字段 | 说明 |
|------|------|
| `value` | satoshi 数量 |
| `scriptPubKey` | 锁定脚本，定义花费条件 |

---

## locktime

- `locktime = 0`：立即有效（仍须入块）
- `< 500000000`：区块高度锁定
- `≥ 500000000`：UNIX 时间戳锁定

配合 `OP_CHECKLOCKTIMEVERIFY`（CLTV）实现定时支付。

---

## txid vs wtxid

| | 计算方式 | 用途 |
|--|----------|------|
| **txid** | 对不含 witness 的序列化做 hash256 | 区块内 Merkle 树、OutPoint 引用 |
| **wtxid** | 对含 witness 的完整序列化做 hash256 | SegWit 后 peer 传播、防 witness 篡改 |

SegWit 引入 **tx witness** 与交易本体分离，修复**交易延展性**（第三方可改签名不改 txid）。

---

## 生命周期

```
constructed → signed → broadcast → mempool → mined → confirmed
                  ↓
              rejected (invalid / too low fee / orphan)
```

**孤儿交易（orphan）：** input 引用的 UTXO 尚不可见（父交易未确认），暂存 orphan pool。

---

## bitcoin-cli 查看交易

```bash
# testnet 示例
bitcoin-cli -testnet getrawtransaction <txid> true
bitcoin-cli -testnet decoderawtransaction <hex>
```

`verbose=true` 返回 JSON，含 `vin`、`vout`、`confirmations`。

---

## 本课小结

- 交易 = inputs 消费 UTXO + outputs 创建 UTXO
- SegWit 分离 witness，产生 wtxid
- sequence / locktime 支持 RBF 与时间锁

# 第 06 课 · Bitcoin Script 入门

> 栈式、逆波兰、非图灵完备——故意保持简单。
>
> **作者：曹宇春**

---

## 执行模型

Bitcoin Script 是**栈式语言**，**无循环**，执行步数上限 **201**（`MAX_OPS_PER_SCRIPT` 等限制）。

每笔 input 验证时执行：

```
<scriptSig>  <scriptPubKey>  →  OP执行 → 栈顶为 true 则通过
```

实际 SegWit 中 scriptSig 常为空，签名在 **witness** 中，scriptPubKey 为 `OP_0 <pubKeyHash>` 等。

---

## 常用操作码

| Opcode | 作用 |
|--------|------|
| `OP_DUP` | 复制栈顶 |
| `OP_HASH160` | RIPEMD160(SHA256(x)) |
| `OP_EQUALVERIFY` | 相等则继续，否则失败 |
| `OP_CHECKSIG` | 验证 ECDSA 签名 |
| `OP_CHECKMULTISIG` | M-of-N 多签 |
| `OP_CHECKLOCKTIMEVERIFY` | 绝对时间锁 |
| `OP_CHECKSEQUENCEVERIFY` | 相对时间锁 |

---

## P2PKH 标准脚本（最常见）

**scriptPubKey（锁定）：**

```
OP_DUP OP_HASH160 <pubKeyHash> OP_EQUALVERIFY OP_CHECKSIG
```

**scriptSig（解锁，legacy）：**

```
<signature> <pubKey>
```

**执行流程：**

1. 压入签名和公钥
2. `OP_DUP` 复制公钥
3. `OP_HASH160` 得 pubkey hash
4. 与 scriptPubKey 中的 hash 比较
5. `OP_CHECKSIG` 验证签名对交易摘要有效

---

## 脚本类型一览

| 类型 | scriptPubKey 特征 | 地址前缀 |
|------|-------------------|----------|
| P2PKH | `DUP HASH160 ... CHECKSIG` | `1...` |
| P2SH | `HASH160 <scriptHash> EQUAL` | `3...` |
| P2WPKH | `0 <20-byte-hash>` | `bc1q...` |
| P2WSH | `0 <32-byte-hash>` | `bc1q...`（32 字节） |
| P2TR | `1 <32-byte-key>` | `bc1p...` |

---

## 故意非图灵完备

| 设计 | 原因 |
|------|------|
| 无循环 | 验证时间可预测，防 DoS |
| 无状态 | 每笔独立验证 |
| 简单栈 | 实现简单，共识风险低 |

复杂逻辑用 **P2SH / Taproot script path** 藏在哈希承诺后。

---

## 调试工具

- [script.bitbom.com](https://script.bitbom.com/) — 在线脚本调试
- `bitcoin-cli decodescript <hex>`
- `bitcoin-cli testmempoolaccept '["<rawtx>"]'` — 测试是否被 mempool 接受

---

## 本课小结

- Script = 定义 UTXO 花费条件的谓词
- P2PKH：`签名 + 公钥` 解锁 `CHECKSIG`
- 非图灵完备是安全特性，不是缺陷

# 第 07 课 · 区块与区块链结构

> 区块头 80 字节，承载 PoW 与 Merkle 承诺。
>
> **作者：曹宇春**

---

## 区块组成

```
Block
├── Header    (80 bytes, 固定)
└── Transactions[]  (var, 含 coinbase 为首笔)
```

---

## 区块头字段（80 字节）

| 字段 | 大小 | 说明 |
|------|------|------|
| version | 4 | 版本，软分叉信号相关 |
| prev_block_hash | 32 | 前一区块头 hash256（小端） |
| merkle_root | 32 | 本块交易 Merkle 根 |
| timestamp | 4 | UNIX 时间 |
| bits | 4 | 压缩编码的难度 target |
| nonce | 4 | 矿工穷举的随机数 |

```
hash256(header) < target  →  区块有效
```

---

## Coinbase 交易

每个区块**第一笔**必须是 coinbase：

- **唯一**允许没有 input（prev_txid 全 0）的交易
- output = **block subsidy**（补贴）+ 本块所有交易 **fee**
- 可含任意 `coinbase` 数据（≤ 100 字节），中本聪创世块含泰晤士报标题

**Subsidy 减半：** 每 210,000 块减半，从 50 BTC 始，现 3.125 BTC（2024 第四次减半后）。

---

## 区块大小限制

- **Legacy 上限：** 1 MB（实际按 weight 计）
- **SegWit 后：** 用 **weight** 计量，理论最大约 **4M weight units**（约 2–4 MB 视 witness 比例）
- **Weight** = `3 × stripped_size + total_size`

---

## 链的选择规则

节点跟随 **最多累计工作量（chainwork）** 的链，而非简单「最长块数」。

```
chainwork = Σ(2^256 / (target_i + 1))  对链上各块
```

分叉时，诚实算力占优的链最终胜出（概率意义上）。

---

## 区块浏览

```bash
bitcoin-cli getblock <blockhash> 2    # verbose JSON
bitcoin-cli getblockheader <hash>
bitcoin-cli getblockchaininfo
```

浏览器：[mempool.space](https://mempool.space)、[blockstream.info](https://blockstream.info)

---

## 本课小结

- 区块头 80 字节 + 交易列表
- Coinbase 产出新币 + 手续费
- 有效链 = 累计工作量最大 + 全部共识规则通过

# 第 08 课 · PoW 与难度调整

> 工作量证明如何锚定安全；难度如何维持 10 分钟出块。
>
> **作者：曹宇春**

---

## PoW 在解什么问题

在开放 P2P 网络中，如何在没有身份系统的情况下：

1. **公平选举**出块者
2. 让**改历史**的成本 ∝ 被改之后所有块的重算成本
3. 使**诚实算力 majority** 时，链稳定扩展

比特币的答案：**算 hash256(header) < target**，谁先找到 nonce（及 extraNonce 等）谁出块。

---

## Target 与 bits

**target** 是一个 256 位整数，哈希值必须**数值上小于**它。

`bits` 字段压缩存储 target（4 字节），类似浮点记法。

```
难度 ∝ 1 / target
```

target 越小 → 难度越高 → 平均需要更多 hash 尝试。

---

## 难度调整（每 2016 块）

约每 **2 周**（2016 × 10 分钟）调整一次：

```
new_target = old_target × (实际用时 / 预期用时)
```

- 预期用时 = 2016 × 600 秒
- 实际用时 = 这 2016 个块的真实跨度
- 调整幅度限制在 **4 倍**以内（单周期最多 ×4 或 ÷4）

**结果：** 长期平均出块间隔 ≈ **10 分钟**，与全网算力规模无关。

---

## 矿工经济学

```
收入 = block_subsidy + Σ(transaction fees in block)
成本 = 电费 + 设备折旧 + 运维 + 机会成本
```

减半使 subsidy 下降，**手续费占比**长期趋势上升。

---

## 算力与安全

| 攻击 | 条件 | 后果 |
|------|------|------|
| 51% 攻击 | 控制 >50% 算力 | 可双花已确认交易（需深度重组）、审查交易 |
| 自私挖矿 | 策略性 withholding | 收益优化，非完全双花 |
| 芬尼攻击 | 0-conf 场景 | 商户即时确认风险 |

**注意：** 51% **不能**凭空造币、不能偷别人 UTXO（仍需私钥），主要威胁是**重组 + 双花**。

---

## 挖矿工程（了解）

- **矿池：** Stratum 协议，矿工提交 share，池子分配奖励
- **ASIC：** SHA-256 专用芯片，GPU/CPU 已无竞争力
- **extraNonce：** 在 coinbase 中改数据，扩展搜索空间（nonce 仅 32 bit）

---

## 本课小结

- PoW = 用可验证的计算成本竞争出块权
- 难度调整维持 10 分钟出块
- 安全假设：诚实算力 > 50%

# 第 09 课 · 共识规则与分叉

> 全节点验证什么；软分叉与硬分叉的本质区别。
>
> **作者：曹宇春**

---

## 全节点验证清单

对新交易（mempool）：

- [ ] 语法正确、大小合法
- [ ] input 引用的 UTXO 存在且未花费
- [ ] 脚本验证通过（签名有效）
- [ ] Σinput ≥ Σoutput，无负值
- [ ] 不违反共识规则（locktime、sequence、weight…）

对新区块：

- [ ] 区块头 PoW 有效
- [ ] 第一笔为合法 coinbase，subsidy 不超过允许值
- [ ] 所有交易通过上述验证
- [ ] Merkle root 正确
- [ ] 时间戳、难度符合规则

**共识 = 所有全节点跑同一套代码规则。**

---

## 软分叉（Soft Fork）

**收紧规则**，新区块对旧节点仍「看起来合法」（或旧节点略宽松）。

- 旧节点可能认为新区块有效，但**不验证新语义**
- 需多数算力升级才能激活
- 例：P2SH（2012）、CLTV、SegWit（BIP-9/141）、Taproot

**激活机制：** BIP-9 版本位、Speedy Trial（Taproot）等。

---

## 硬分叉（Hard Fork）

**放宽或改变规则**，旧节点不认新区块 → **链分裂**。

- 例：2017 BCH（区块大小争议）、BSV
- 需生态各方协调，否则永久两条链、两种资产

---

## 常见分叉类型

| 类型 | 原因 | 持久性 |
|------|------|--------|
| 临时分叉 | 两矿工同时出块 | 下一块后自然解决 |
| 硬分叉 | 规则不兼容 | 永久分裂 |
| 软分叉 | 规则收紧 | 不分裂（成功时） |

---

## 用户视角

- **软分叉成功：** 用户无感，用升级后的钱包即可
- **硬分叉：** 持有私钥者在分叉点可能获得**两条链上的币**（若两边都有市场）

**运行全节点** = 自己投票认可哪条链的规则集。

---

## BIP 流程简述

```
Idea → BIP 草案 → 参考实现 PR → 社区讨论 → 矿工/节点升级 → 激活
```

重要 BIPs 存档：[github.com/bitcoin/bips](https://github.com/bitcoin/bips)

---

## 本课小结

| | 软分叉 | 硬分叉 |
|--|--------|--------|
| 规则 | 收紧 | 改变/放宽 |
| 兼容性 | 向前兼容 | 不兼容 |
| 链 | 通常一条 | 可能分裂 |

# 第 10 课 · 地址类型演进

> 从 P2PKH 到 Bech32 再到 Taproot，编码的是 script 承诺。
>
> **作者：曹宇春**

---

## 地址不是账户

**地址**是 `scriptPubKey` 的人类可读编码，方便转账时指定锁定条件。

链上不存在「地址余额」，只有 UTXO 的 `scriptPubKey` 匹配。

---

## 演进时间线

```
2009  P2PKH      1...
2012  P2SH       3...
2017  SegWit     bc1q... (P2WPKH / P2WSH)
2021  Taproot    bc1p... (P2TR)
```

---

## P2PKH（Pay to Public Key Hash）

```
scriptPubKey: OP_DUP OP_HASH160 <20-byte> OP_EQUALVERIFY OP_CHECKSIG
地址: Base58Check, 版本字节 0x00 → 主网以 1 开头
```

---

## P2SH（Pay to Script Hash）

把**复杂赎回脚本**先 HASH160，链上只暴露脚本哈希。

```
scriptPubKey: OP_HASH160 <20-byte scriptHash> OP_EQUAL
赎回时 scriptSig: <sig> <redeemScript>
```

用例：多签、嵌套 SegWit（P2SH-P2WPKH）。

---

## Bech32 / Bech32m 编码

SegWit 与 Taproot 使用 **Bech32**（SegWit）或 **Bech32m**（Taproot v1+）：

- 小写 `bc1` 主网前缀
- 内置校验和，防输错
- 无大小写混用歧义（Bech32 限制）

---

## 生成地址（概念）

```text
私钥 → 公钥 → HASH160(pubkey) → Base58Check → P2PKH 地址
私钥 → 公钥 → Witness program → Bech32 encode → bc1q...
```

使用库完成，不要手写 Base58。

---

## 本课小结

- 地址 = scriptPubKey 的编码，不是链上原生概念
- P2PKH / P2SH / P2WPKH / P2TR 对应不同锁定逻辑
- 新钱包应默认 **bc1q** 或 **bc1p**

# 第 11 课 · SegWit 与见证数据

> BIP-141：修复延展性、提高有效容量、为闪电铺路。
>
> **作者：曹宇春**

---

## 解决的问题

### 1. 交易延展性（Malleability）

第三方可在签名编码上做微调，**改变 txid 但不无效交易**。  
对闪电网络等二层，需要稳定的 txid 作为承诺。

### 2. 区块容量

签名等数据移入 **witness**，按较低 weight 计费 → 同等 weight 下容纳更多交易。

### 3. 脚本版本升级

见证版本字段（`0x00` + program）便于未来新脚本类型。

---

## 结构变化

SegWit 交易在序列化中带 **marker（0x00）flag（0x01）**：

- **stripped transaction：** 不含 witness，用于算 **txid**
- **full transaction：** 含 witness，用于算 **wtxid**

每个 input 的 witness 栈独立存放签名、公钥等。

---

## P2WPKH（原生 SegWit 单签）

```
scriptPubKey: OP_0 <20-byte pubkey hash>
witness: <signature> <pubkey>
scriptSig: (empty)
```

地址：`bc1q...`（42 字符左右）

---

## P2SH-P2WPKH（嵌套 SegWit）

兼容不支持 bc1 的老地址：外层 P2SH，赎回脚本为 `0x0014<pubkeyhash>`。

```
发送方看到 3... 地址，实际 witness 在内部解锁
```

---

## Weight 与 Fee 计算

```
weight = 3 × stripped_size + total_size
vsize = weight / 4  (虚拟字节)
fee rate 通常以 sat/vB 报价
```

witness 字节在 weight 中「打折」，鼓励 SegWit 采用。

---

## 激活回顾

- BIP-141 定义 SegWit
- 2017 年通过 **BIP-9** 版本位激活（UASF / SegWit2x 争议背景）
- 现主网 SegWit 交易占绝大多数

---

## 本课小结

- SegWit = witness 分离 + weight 计费
- txid 稳定 → 二层可行
- 开发应默认构造 **native SegWit** 交易

# 第 12 课 · Taproot 与 Schnorr

> BIP-341/342：默认最简路径、隐私与多签聚合。
>
> **作者：曹宇春**

---

## 动机

| 问题 | Taproot 方案 |
|------|--------------|
| 复杂脚本链上占空间 | 默认只暴露单签，复杂逻辑走 script path |
| ECDSA 签名大 | Schnorr 64 字节，可聚合 |
| 多签链上明显 | MuSig 类方案可看起来像单签 |

---

## 密钥聚合（概念）

多个参与方协作生成**单一公钥**（Taproot internal key），链上只见一个 `bc1p` 地址。

花费时：

- **Key path：** 合作签名，不暴露脚本
- **Script path：** 揭示 Merkle 树中某叶子脚本 + 满足条件

---

## P2TR 输出

```
scriptPubKey: OP_1 <32-byte tweaked pubkey>
地址: bc1p... (Bech32m)
```

**Tweak：** 用 Merkle root 等对 internal key 调整，绑定 script 树承诺。

---

## Schnorr 签名（BIP-340）

- 签名 64 字节（r || s）
- **批量验证**效率更高
- 支持 **BIP-322** 消息签名等扩展

Bitcoin Core 0.21+ 支持 Taproot；2021 年 11 月区块 709,632 激活。

---

## Key Path vs Script Path

```
Key Path（常见）:
  witness: <schnorr_sig>
  链上看起来像普通单签转账

Script Path（条件分支）:
  witness: <stack items> <script> <control block>
  control block 含 Merkle 路径证明
```

---

## 开发注意

- 使用支持 Taproot 的库（Bitcoin Core 22+、rust-bitcoin 0.29+、bitcoinjs-lib 6+）
- 多签优先评估 **MuSig2** 而非裸 `OP_CHECKMULTISIG`（费、隐私）
- 测试网 / signet 先验证

---

## 本课小结

- Taproot = Schnorr + MAST（Merklized Alternative Script Tree）
- 默认 key path 最省空间、隐私最好
- 比特币脚本演进终点（目前）是 **bc1p**

# 第 13 课 · P2P 网络与节点类型

> 交易和区块如何在互联网上传播。
>
> **作者：曹宇春**

---

## 网络拓扑

比特币是 **无中心 P2P 网络**：

- 节点间 TCP 连接（默认 8333 mainnet）
- 无全局目录服务器，靠 **DNS seed**、硬编码种子节点发现 peer
- 消息二进制协议，`magic bytes` 区分 mainnet/testnet

---

## 主要消息类型

| 消息 | 作用 |
|------|------|
| `version` / `verack` | 握手、协商版本 |
| `inv` | 清单：有新 tx/block 的哈希 |
| `getdata` | 请求具体 tx/block |
| `tx` | 传播交易 |
| `block` | 传播区块 |
| `headers` | 传播区块头（轻同步） |
| `addr` | peer 地址 gossip |

**传播模型：** 新交易 → inv → getdata → tx；节点验证后入 mempool，继续转发。

---

## 节点类型

| 类型 | 特点 |
|------|------|
| **全节点** | 完整验证，转发区块/交易 |
| **剪枝节点** | 删旧块数据，保留近期 + UTXO |
| **仅网络中继** | 部分节点限制 RPC、不服务轻钱包 |
| **SPV 客户端** | 只拉 headers + Merkle proof |
| **挖矿节点** | 全节点 + 矿池 Stratum 连接 |

---

## Mempool（内存池）

每个全节点本地维护未确认交易集合：

- 按 **ancestor fee rate** 等排序供矿工选题
- **容量限制** → 低费交易被驱逐
- **冲突检测** → 双花 input 拒绝

```bash
bitcoin-cli getmempoolinfo
bitcoin-cli getrawmempool true
```

---

## 隐私与网络层

- P2P 流量未默认加密（v2 transport BIP-324 逐步推广加密）
- 链分析可通过 **节点连接模式、地址重用** 推断身份
- 运行自己的全节点可改善隐私（不泄露查询地址给第三方 API）

---

## 本课小结

- 比特币靠 P2P gossip 传播 tx/block
- 全节点 = 验证 + 中继 + mempool
- 开发可依赖自建节点 RPC，而非只信区块浏览器 API

# 第 14 课 · 全节点验证与同步

> Initial Block Download、链状态、剪枝与 assumevalid。
>
> **作者：曹宇春**

---

## 同步阶段

新全节点启动后典型流程：

```
1. 发现 peers（DNS seeds）
2. 下载 headers（从 genesis 到 tip，快）
3. 下载 blocks（IBD，慢，主要瓶颈）
4. 验证每笔交易、更新 UTXO set
5. 进入 steady state，监听新区块
```

**IBD（Initial Block Download）：** 主网数 TB 级历史，数小时到数天取决于硬件与带宽。

---

## 验证层次

| 层次 | 内容 |
|------|------|
| Header 验证 | PoW、时间、难度 |
| 交易验证 | 脚本、UTXO、共识 |
| 索引 | 可选 txindex 支持任意 txid 查询 |

默认 Bitcoin Core **不开启 txindex**（省空间），按 txid 查询需 block height 提示或第三方索引。

---

## 剪枝模式（prune）

```ini
# bitcoin.conf
prune=550   # MB，最小约 550
```

- 只保留近期区块 + UTXO set
- 仍是**全验证**节点
- 不能服务历史区块给 peer（但自己安全）

---

## assumevalid 与 checkpoints

为加速 IBD，Bitcoin Core 内置 **assumevalid** 区块哈希：对该块之前的脚本**跳过验证**（仍验证 PoW 和结构），信任发布版本。

**不是信任第三方链：** 仍验证 headers 链与 PoW；若 assumevalid 块无效，后续会失败。

---

## 链状态查看

```bash
bitcoin-cli getblockchaininfo
# chain, blocks, headers, verificationprogress, initialblockdownload

bitcoin-cli getchaintxstats
bitcoin-cli gettxoutsetinfo   # UTXO set 统计，较慢
```

---

## 为什么开发者应跑自己的节点

| 依赖第三方 API | 自建节点 |
|----------------|----------|
| 信任对方返回 | 自己验证 |
| 泄露查询隐私 | 本地 RPC |
| 限流 / 宕机 | 自主可控 |
| 测试网同样适用 | regtest 秒级出块 |

---

## 本课小结

- 全节点同步 = headers + blocks + UTXO 验证
- 剪枝可省磁盘，不牺牲安全（对自己）
- regtest/signet 用于开发，mainnet 用于生产验证

# 第 16 课 · 构造与签名交易

> 从 listunspent 到 signrawtransactionwithwallet 再到 sendrawtransaction。
>
> **作者：曹宇春**

---

## 高层 API（钱包已管理密钥）

```bash
# regtest 环境，已有余额

# 1. 新建收款地址
DEST=$(bitcoin-cli -regtest getnewaddress)

# 2. 发送（内部完成选币、构造、签名、广播）
TXID=$(bitcoin-cli -regtest sendtoaddress $DEST 0.01)
echo $TXID

# 3. 出块确认
bitcoin-cli -regtest generatetoaddress 1 $(bitcoin-cli -regtest getnewaddress)

# 4. 查看交易
bitcoin-cli -regtest gettransaction $TXID
```

---

## 底层流程（理解协议）

```
1. listunspent     → 选 input UTXO
2. createrawtransaction → 指定 inputs + outputs
3. fundrawtransaction   → 自动选币、加找零、算 fee（可选）
4. signrawtransactionwithwallet / withkey
5. testmempoolaccept    → 预检
6. sendrawtransaction   → 广播
```

### 示例

```bash
# 假设已有 UTXO
UTXO=$(bitcoin-cli -regtest listunspent 1 | jq '.[0]')
TXID=$(echo $UTXO | jq -r '.txid')
VOUT=$(echo $UTXO | jq -r '.vout')

CHANGE=$(bitcoin-cli -regtest getrawchangeaddress)
DEST=$(bitcoin-cli -regtest getnewaddress)

RAW=$(bitcoin-cli -regtest createrawtransaction \
  "[{\"txid\":\"$TXID\",\"vout\":$VOUT}]" \
  "{\"$DEST\":0.001,\"$CHANGE\":0.0089}")

FUNDED=$(bitcoin-cli -regtest fundrawtransaction $RAW)
SIGNED=$(bitcoin-cli -regtest signrawtransactionwithwallet \
  $(echo $FUNDED | jq -r '.hex'))

bitcoin-cli -regtest sendrawtransaction \
  $(echo $SIGNED | jq -r '.hex')
```

---

## 签名哈希（SIGHASH）

签名不只签整个交易，通过 **SIGHASH 标志**指定签哪些 input/output：

| 标志 | 含义 |
|------|------|
| SIGHASH_ALL | 默认，签所有 input/output |
| SIGHASH_NONE | 不承诺 output |
| SIGHASH_SINGLE | 只承诺同 index 的 output |

开发一般只用 `SIGHASH_ALL`。

---

## 费率估算

```bash
bitcoin-cli -regtest estimatesmartfee 6
bitcoin-cli -regtest getmempoolinfo
```

生产环境参考 [mempool.space](https://mempool.space) 的 sat/vB 推荐。

---

## 库构造（JavaScript 概念）

```javascript
// bitcoinjs-lib 伪代码
import * as bitcoin from 'bitcoinjs-lib'

const psbt = new bitcoin.Psbt({ network: bitcoin.networks.regtest })
psbt.addInput({ hash: txid, index: vout, witnessUtxo: { script, value } })
psbt.addOutput({ address, value })
psbt.signInput(0, keyPair)
psbt.finalizeAllInputs()
const tx = psbt.extractTransaction()
```

现代流程优先 **PSBT**（下一课），而非裸 `createrawtransaction`。

- 构造 = 选 UTXO + outputs + fee
- 签名 = 对 sighash 用私钥 ECDSA/Schnorr
- 广播前用 `testmempoolaccept` 预检

# 第 17 课 · PSBT 与多重签名

> BIP-174：部分签名比特币交易，多签与硬件钱包的标准格式。
>
> **作者：曹宇春**

---

## 为什么需要 PSBT

| 场景 | 难点 | PSBT 解决 |
|------|------|-----------|
| 冷/热分离 | 离线机无 UTXO 信息 | 携带 prevtx 数据 |
| 多签 | 多方分别签名 | 增量签名状态 |
| 硬件钱包 | 主机不可信 | 设备只签展示过的摘要 |

**PSBT** = 可传输、可部分签名的交易表示（Base64 或二进制）。

---

## PSBT 角色

```
Creator → Updater → Signer(s) → Finalizer → Extractor → Broadcast
```

- **Creator：** 建空白 PSBT
- **Updater：** 加 input/output、UTXO 详情
- **Signer：** 各签各的 input
- **Finalizer：** 合成 witness/scriptSig
- **Extractor：** 导出裸 raw tx

---

## 2-of-3 多签概念

```bash
# Bitcoin Core 创建 descriptor 钱包（简化示例）
bitcoin-cli -regtest createwallet "multisig" true true "" true true

# 使用 sortedmulti descriptor
# wsh(sortedmulti(2, key1, key2, key3))
```

链上 `scriptPubKey` 含 `OP_2 <pk1> <pk2> <pk3> OP_3 OP_CHECKMULTISIG`。

花费需任意 2 方签名。裸多签**费大、隐私差**；新项目考虑 **Taproot + MuSig2**。

---

## PSBT 工作流（CLI 概念）

```bash
# 1. 创建 PSBT
bitcoin-cli -regtest walletcreatefundedpsbt \
  '[]' '[{"<addr>":0.01}]'

# 2. 签名
bitcoin-cli -regtest walletprocesspsbt <psbt>

# 3. 完成并广播
bitcoin-cli -regtest finalizepsbt <psbt>
bitcoin-cli -regtest sendrawtransaction <hex>
```

多钱包场景用 `joinpsbts` 合并签名。

---

## 硬件钱包流程

1. 主机构造 PSBT，只含必要 UTXO 信息
2. USB/NFC 连硬件，屏幕显示金额/地址
3. 用户确认，设备签名
4. 主机 finalize + broadcast

**绝不**把助记词输入主机。

---

## 本课小结

- PSBT = 多签/离线的工业标准
- 2-of-3 适合公司金库、继承规划
- 优先 Taproot MuSig 而非 legacy multisig

# 第 18 课 · 钱包标准 BIP-32/39/44

> HD 钱包、助记词与派生路径——库实现的基础。
>
> **作者：曹宇春**

---

## BIP-39：助记词

- 128–256 bit 熵 → 映射为 **12/15/18/21/24** 个英文词
- 含校验和，错一个词大概率无法恢复
- **PBKDF2** 从助记词 + 可选 passphrase 导出 seed

```
助记词 → seed (512 bit) → BIP-32 主密钥
```

**passphrase：** 可选第 25 个词，防助记词泄露被直接花掉（需记牢）。

---

## BIP-32：分层确定性密钥

从 seed 导出树状密钥：

```
m / purpose' / coin_type' / account' / change / address_index
```

- `m`：主节点
- `'`：硬化派生（hardened），子公钥不能推兄弟私钥
- **xpub**：扩展公钥，可派生收款地址但不能花（无私钥）

```bash
bitcoin-cli -regtest listdescriptors
# 现代 Bitcoin Core 默认 descriptor 钱包
```

---

## BIP-44 / 49 / 84 / 86：coin type 与脚本类型

| BIP | purpose | 路径示例 | 地址类型 |
|-----|---------|----------|----------|
| 44 | 44' | m/44'/0'/0'/0/0 | P2PKH (1...) |
| 49 | 49' | m/49'/0'/0'/0/0 | P2SH-P2WPKH (3...) |
| 84 | 84' | m/84'/0'/0'/0/0 | P2WPKH (bc1q...) |
| 86 | 86' | m/86'/0'/0'/0/0 | P2TR (bc1p...) |

`0'` = Bitcoin mainnet；testnet 常用 `1'`。

---

## Descriptor 钱包（现代标准）

Bitcoin Core 22+ 默认 **descriptor wallet**，用字符串描述脚本：

```
pkh(...)           # P2PKH
wpkh(...)          # P2WPKH
tr(...)            # Taproot
wsh(sortedmulti(...))  # 多签
```

```bash
bitcoin-cli -regtest listdescriptors true
bitcoin-cli -regtest getdescriptorinfo "wpkh(...)"
```

优于老的 BIP-44 账户模型：表达力强、可导入导出明确。

---

## 实现注意

| 规则 | 原因 |
|------|------|
| 每笔收款用新地址 | 隐私 |
| 离线生成助记词 | 防泄露 |
| 不用脑记词钱包 | 熵不够 |
| 备份 passphrase | 丢了 seed 变砖 |

---

## 本课小结

- BIP-39 → 人类可读 seed
- BIP-32 → 一棵树派生无数地址
- BIP-84/86 → 现代 bc1q / bc1p 路径
- 生产用 descriptor + PSBT

# 第 19 课 · 安全威胁与攻击面

> 协议层、实现层、应用层分别怕什么。
>
> **作者：曹宇春**

---

## 协议层威胁

| 威胁 | 说明 | 缓解 |
|------|------|------|
| 51% 攻击 | 算力 majority 重组 | 多确认、算力去中心化 |
| 双花 | 同一 UTXO 花两次 | 全节点拒绝；商户多确认 |
| 芬尼攻击 | 0-conf 商户即时交付 | 不接受未确认大额 |
| 时间扭曲 | 操纵 timestamp 影响难度 | 共识限制时间偏差 |

---

## 钱包 / 应用层威胁

| 威胁 | 说明 | 缓解 |
|------|------|------|
| 助记词泄露 | 钓鱼、木马、云备份 | 硬件钱包、离线签名 |
| 地址替换 | 剪贴板恶意软件 | 硬件屏确认、地址校验 |
| 假钱包 App | 偷词、偷私钥 | 官方渠道、开源可审计 |
| 交易所被黑 | 托管风险 | 自持密钥 |
| 供应链攻击 | 依赖库后门 | 校验 checksum、复现构建 |

---

## 开发层陷阱

| 陷阱 | 后果 |
|------|------|
| 错误 SIGHASH | 签名可被复用到别的交易 |
| 整数溢出（历史） | 比特币已用 satoshi 整数 |
| 重用 nonce（ECDSA） | 可推私钥（2010 PlayStation 类事故） |
| 不验证 output 金额 | 被 fee 攻击 |
| 信任第三方 fee 估算 | 多付或 stuck |

**Schnorr/MuSig** 需注意 nonce 生成必须密码学安全随机（或用 RFC6979 类确定性方案）。

---

## 隐私与链分析

- 地址重用 → 关联所有交易
- KYC 交易所入金 → 链上地址 deanonymize
- CoinJoin / PayJoin 提高隐私（钱包如 Wasabi、Samourai 原理）
- **Taproot** 默认让多签与单签看起来相似

---

## 安全开发清单

```
□ 在 regtest/signet 充分测试
□ 大额用 PSBT + 硬件签名
□ 核对 output 地址与金额（人机双重）
□ 不记录助记词到日志/数据库
□ 多签 + 地理分散备份
□ 监控异常出块 / 深度重组（交易所）
```

---

## 本课小结

- 比特币协议久经考验；风险多在**密钥管理与应用**
- 0-conf 不适合高价值交付
- 开发者对签名语义和金额校验负全责

# 第 20 课 · 闪电网络原理

> 链下通道、HTLC、路由——比特币二层支付。
>
> **作者：曹宇春**

---

## 解决什么问题

链上比特币：

- ~10 分钟确认
- 主网 fee 波动
- 隐私性有限

**闪电网络（LN）** 在链下建立支付通道，即时、小额、低费；最终仍锚定比特币主链结算。

---

## 支付通道（Payment Channel）

两人各锁一笔资金到 **2-of-2 多签** UTXO：

```
链上: Funding TX → 2-of-2 output
链下: 不断更新「最新余额分配」签名状态
关闭: 合作关闭（一笔 tx）或单方强制关闭（带延迟惩罚）
```

**关键：** 最新状态签名可链上执行；广播旧状态会被 **penalty** 罚没保证金。

---

## HTLC（Hash Time Locked Contract）

跨通道路由的核心：

```
「若你知道 preimage R，使得 hash(R)=H，则在时间 T 前你可领取；
 否则收款方超时收回。」
```

多跳路径上形成 **哈希锁链**，原子性：要么全成功，要么全失败。

---

## 路由

A → B → C → D 付款：

- A 不必信任 B、C，密码学 + 经济激励保证
- 每跳收取微小 routing fee
- 路径发现：gossip 网络广播 channel graph

---

## 实现与生态

| 实现 | 语言 |
|------|------|
| LND | Go |
| Core Lightning (CLN) | C |
| Eclair | Scala |

规范：**BOLT**（Basis of Lightning Technology）系列文档。

```bash
# 实验（需同步 testnet/signet + 配置）
# lnd / lightning-cli getinfo
```

---

## 限制与风险

- 需**在线**收 HTLC（接收方）
- **入金流动性**：通道 inbound capacity
- 通道对手方可 **force close**
- 实现漏洞历史上有真实损失（需跟版本）

---

## 本课小结

- LN = 链下状态更新 + 链上锚定 + HTLC 路由
- 适合高频小额；大额仍常用链上多确认
- 深入读：[Mastering the Lightning Network](https://github.com/lnbook/lnbook)

|------|------|
| CoinJoin | 多用户合并输入/output 模糊关联 |
| PayJoin (P2EP) | 付款同时接收，打破启发式 |
| Silent Payments (BIP-352) | 静态地址，链上不可关联 |
| Dandelion++ | 交易传播隐私 |

---

## 侧链与联邦链

| 项目 | 模型 |
|------|------|
| Liquid | Blockstream 联邦侧链，机密资产 |
| Rootstock (RSK) | 合并挖矿 EVM 侧链 |
| Fedimint | 社区联邦 Chaumian mint |

与主链**双向锚定**或**信任联邦**——安全模型不同。
