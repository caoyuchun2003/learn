## 1. 问题定义（一句话）

**帮助中小教培机构的老师，在自有题库基础上更快完成「录题 → 变式扩充 → 按知识点组卷」——AI 只出草稿，人审后才入库。**

---

## 2. 目标用户 & 核心场景

### 目标用户（MVP 只服务一类）

| 用户 | 是否 MVP | 说明 |
|------|----------|------|
| **学科老师 / 教研员** | ✅ 核心 | 录题、审核 AI 题、组卷 |
| 教务管理员 | ⚪ 次要 | 后期：权限、年级班级 |
| 学生 | ❌ MVP 不做 | 第二期再考虑在线练习 |

**虚构 persona：**

- **李老师**，初中数学教师，机构 200 人规模  
- 每周需出 2 份练习卷，题库散落于 Excel、Word、微信群文件  
- 会用电脑，不排斥 AI，但不敢直接用 AI 题不审核就给学生  

### 核心场景（高频）

```
场景 A：周练组卷
  周五下午 → 从题库按「一元一次方程」抽 10 题 → 导出 PDF 打印

场景 B：同类题扩充
  讲义里有一道好题 → 希望再生成 5 道同考点变式 → 人工改一改入库

场景 C：新章节目录初始化
  新章节开课 → 批量录入 20 道基础题 → 打知识点标签
```

MVP 优先支持：**场景 B + 场景 A**（B 体现 AI 价值，A 体现闭环）。

---

## 3. 痛点 & 现有替代方案

### 痛点（用户原话 · 虚构访谈）

| # | 原话摘要 | 类型 |
|---|----------|------|
| 1 | 「题都在 Excel 里，组卷要复制粘贴一小时。」 | 事实 |
| 2 | 「网上组卷网站题不准我们教学进度，也不敢用。」 | 事实 |
| 3 | 「ChatGPT 出的题有时答案错，不敢直接给学生。」 | 事实 |
| 4 | 「我们机构应该会买单吧。」 | **假设，待验证** |

### 现有替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Excel / Word 自建 | 自由、无费用 | 检索差、组卷慢、无版本 |
| 公有题库 / 组卷 SaaS | 题量大、组卷快 | 题不符合本校进度；数据在公网 |
| 通用 AI 聊天出题 | 快 | 无题库结构；质量不稳；无审核流 |
| 纸质教辅 | 质量相对稳定 | 不能自定义；无数据 |

### 机会点（推断）

机构老师要的是：**私有题库 + 结构化 + AI 辅助但可控**——不是又一个公网刷题平台。

---

## 4. 方案建议 & 差异化

### 产品定位（虚构）

**「智练题库」—— 老师的私有题库工作台，AI 是助教不是主角。**

### 差异化（相对公有 SaaS + 裸 ChatGPT）

| 维度 | 智练题库（虚构方案） |
|------|----------------------|
| 数据 | 机构私有，可部署内网（远期） |
| AI 角色 | 只生成「待审核草稿」，不入库 |
| 结构 | 题目绑知识点树，可检索、可组卷 |
| 用户 | 先老师，后学生 |

### MVP 功能范围

| # | 功能 | 说明 |
|---|------|------|
| F1 | 题目 CRUD | 单选/多选/填空；题干、选项、答案、解析 |
| F2 | 知识点标签 | 树形或扁平标签，一题多标签 |
| F3 | AI 变式出题 | 输入 1 道样题 → 输出 N 道草稿 → **人审 → 入库** |
| F4 | 按标签组卷 | 选知识点 + 题数 → 生成试卷 → 导出 PDF/Word |

---

## 5. 关键假设 & 验证方法

| # | 假设 | 类型 | 验证方法 | 通过标准 |
|---|------|------|----------|----------|
| H1 | 老师愿意先把题录入系统 | 推断 | 3 位老师试用录 20 题 | 2/3 认为不比 Excel 更烦 |
| H2 | AI 变式题改一改能用 | 推断 | 生成 50 道，老师标注可用率 | 可用率 ≥ 60% |
| H3 | 组卷导出能替代周五手工 | 推断 | 计时对比组卷 | 时间减少 ≥ 50% |
| H4 | 机构愿意为「私有题库」付费 | **假设** | 10 家机构访谈 | ≥ 3 家表达付费意愿 |
| H5 | 学生端不是第一期刚需 | 推断 | 问老师「没有学生端能否先用」 | 多数表示可以 |

**事实（已有材料）：** 老师普遍用 Excel；对 AI 题质量有顾虑。  
**不能编造：** 市场规模、付费转化率、竞品份额——标「待调研」。

## 6. 风险 & 不做清单

### 风险

| 风险 | 等级 | 应对 |
|------|------|------|
| AI 答案错误 | 高 | 强制人审；高亮待核对字段 |
| 录题太麻烦没人用 | 高 | 简化表单；支持批量导入 |
| 与公有题库无差异 | 中 | 强调私有 + 审核流 |
| 版权（教材题） | 中 | 只录自有/原创题；用户协议声明 |
| 范围蔓延 | 中 | 严格 MVP，见不做清单 |

### 不做清单（第一版）

- ❌ 学生 C 端刷题、排行榜、社交  
- ❌ AI 题不经审核直接进库  
- ❌ 全学科全学段（MVP 只选 **一个学科**，如初中数学）  
- ❌ 拍照搜题、直播课  
- ❌ 复杂自适应推题算法  
- ❌ 与任何真实考试/出版社联名叙事

## 7. MVP 建议 & 下一步

### RICE 粗排（教学示范）

| 功能 | Reach | Impact | Confidence | Effort | 建议 |
|------|-------|--------|------------|--------|------|
| F1 录题 | 高 | 高 | 高 | 中 | **P0** |
| F2 知识点 | 高 | 高 | 高 | 中 | **P0** |
| F3 AI 变式 | 中 | 高 | 中 | 高 | **P0**（差异化） |
| F4 组卷导出 | 高 | 高 | 高 | 中 | **P0** |
| 学生在线做题 | 高 | 中 | 低 | 高 | **不做** |
| 机构多租户 | 低 | 中 | 中 | 高 | **延后** |

### 建议迭代顺序

```
迭代 1（2 周）：F1 + F2 — 能录能搜
迭代 2（2 周）：F4 — 能组卷导出（无 AI 也有价值）
迭代 3（2 周）：F3 — AI 变式 + 审核台
```

### 下一步行动（虚构团队）

| 谁 | 做什么 | 何时 |
|----|--------|------|
| 产品 | 找 3 位老师做 H1/H2 访谈 | 本周 |
| 设计 | 录题表单 + 审核台线框图 | 下周 |
| 开发 | 迭代 1 技术方案（见下节） | 下周 |
| 全员 | 定一个学科试点（如初中数学） | 本周 |

## 8. 技术粗评（衔接开发 · 教学用）

> 不写具体代码，只列开发 Context，供 Claude Code 练习使用。

| 维度 | 建议 |
|------|------|
| 实现路径 | 后端 REST API + 简单管理后台；AI 调用独立 service |
| 核心表 | `questions`、`tags`、`papers`、`paper_items`、`ai_drafts` |
| AI 风险 | 答案校验可二期加「可选二次模型核对」；一期靠人审 |
| 依赖 | 大模型 API；PDF 导出库 |
| 验证项 | 录题 10 道 < 15 分钟；组卷 < 3 分钟；AI 草稿必进审核表 |

### 复制到 Claude Code 的开发约束（示例）

```
【背景】教学案例「智练题库」迭代 1：题目 CRUD + 知识点标签
【验收】
  - 可创建单选/填空题，绑 1+ 标签
  - 可按标签筛选题目列表
  - API 有基础测试
【约束】
  - 最小改动，不引入学生端
  - 不做 AI 出题（迭代 3 再做）
  - 不要改 .env
【技术栈】（练习时可自定，如 Node + SQLite）
```
